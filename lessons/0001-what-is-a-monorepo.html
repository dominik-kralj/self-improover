What is a monorepo?

Understanding the structure before you scaffold issue #8

The problem it solves

Self-Improover has two separate programs: a React frontend and a Hono backend. You could put them in two completely separate git repositories — a polyrepo. Many teams do.

But with two repos, every shared concern doubles: two CI configs, two sets of ESLint rules, two TypeScript setups to keep in sync. When you want to make a change that touches both sides — say, changing the shape of a Check-in — you open two repos, make two PRs, and pray they land together.

A monorepo puts both programs in one git repository, each in its own folder called a workspace package. You get one commit history, one lint config, one place to look.

One repo ≠ one program. A monorepo is not a monolith. The frontend and backend are still completely separate Node.js packages — they just live in the same git repo. They are deployed separately, built separately, and have their own package.json files.

The structure for Self-Improover

Here is exactly what issue #8 will produce:

self-improover/ ← one git repo ├── package.json ← root: workspaces config + shared scripts ├── tsconfig.base.json ← shared TypeScript settings ├── .eslintrc.js ← shared ESLint rules (issue #9) ├── .prettierrc ← shared Prettier config (issue #9) │ ├── frontend/ │ ├── package.json ← frontend's own deps (React, Chakra UI…) │ ├── tsconfig.json ← extends tsconfig.base.json │ └── src/ │ └── main.tsx │ └── backend/ ├── package.json ← backend's own deps (Hono, Drizzle…) ├── tsconfig.json ← extends tsconfig.base.json └── src/ └── index.ts

How npm workspaces glues it together

The root package.json declares the workspace packages. That is the only special thing about it: 

{
  "name": "self-improover",
  "private": true,
  "workspaces": [
    "frontend",
    "backend"
  ]
}

Running npm install at the root installs all dependencies for both packages in one go. npm hoists shared dependencies to the root node_modules/ and puts package-specific ones inside each folder's node_modules/.

You can run a script in a specific workspace from the root:

# Start only the backend
npm run dev --workspace=backend

# Run lint in both packages
npm run lint --workspaces

TypeScript across packages

Both packages will use TypeScript strict mode. Rather than copy the same settings twice, you write them once in a base config at the root, and each package extends it: 

// tsconfig.base.json (root)
{
  "compilerOptions": {
    "strict": true,
    "target": "ES2022",
    "moduleResolution": "bundler",
    "esModuleInterop": true
  }
}

// frontend/tsconfig.json
{
  "extends": "../tsconfig.base.json",
  "compilerOptions": {
    "lib": ["ES2022", "DOM"],
    "jsx": "react-jsx"
  }
}

// backend/tsconfig.json
{
  "extends": "../tsconfig.base.json",
  "compilerOptions": {
    "lib": ["ES2022"],
    "module": "NodeNext"
  }
}

The frontend needs DOM types (for window, document, etc.). The backend does not — those would be wrong on a Node.js server. This is why the split exists.

What this is not

You may have heard of Turborepo or Nx — tools that add build caching, task orchestration, and remote caching on top of monorepos. You do not need them for Self-Improover. They solve problems that appear at scale (hundreds of packages, long CI times). npm workspaces alone is exactly the right tool here.

The rule of thumb: use plain npm workspaces until you feel the pain Turborepo solves. You will know it when you feel it — and you almost certainly won't on a two-package project.
