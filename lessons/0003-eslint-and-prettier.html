ESLint + Prettier

Two tools. One job each. Never overlap.

The problem they solve

Every codebase develops two kinds of inconsistency over time — things that look different and things that behave dangerously:

  • One file uses 2-space indent, another uses 4. One uses single quotes, another uses double.
  • A variable is declared but never used. A function accepts any and silently breaks at runtime.

The first kind is style — annoying, but harmless. The second kind is correctness — these are bugs waiting to happen.

ESLint and Prettier solve different problems:

Tool Job Examples
Prettier Formatting only indentation, quotes, semicolons, trailing commas
ESLint Correctness only unused vars, any types, broken React hooks
The rule of thumb: If it changes the visual layout of your code but not its meaning — Prettier owns it. If it could be a bug — ESLint owns it.

Why two tools instead of one?

ESLint can enforce formatting rules (it did this for years), but Prettier does it far better: it reprints your entire file from scratch, so the output is always identical no matter what the original looked like. ESLint just reports violations — you still have to fix them manually.

The problem is that if you run both and their opinions differ, ESLint keeps complaining about formatting that Prettier already fixed correctly. That's the conflict — solved by eslint-config-prettier, which disables every ESLint formatting rule so Prettier wins unopposed.

Flat config — the new ESLint format

Until 2024, ESLint used .eslintrc.json — a nested config format that was hard to compose. ESLint 9 replaced it with flat config: a single eslint.config.mjs that exports an array of config objects. Each object can set files, plugins, and rules.

Think of it as a stack of layers. Each layer can add rules, and later layers win:

eslint.config.mjs — layers, top to bottom 1. { ignores } ← global ignore patterns 2. js.configs.recommended ← base JS rules 3. ...tseslint.configs.recommended ← TypeScript rules 4. { files: ["**/*.ts"], rules } ← our overrides (any → error, unused → error) 5. { files: ["frontend/**"] } ← React rules scoped to frontend only 6. prettierConfig ← LAST: strips all formatting rules
Order matters. eslint-config-prettier must be the last item in the array. It works by disabling rules — if something after it re-adds a formatting rule, the conflict returns.

Why React plugins are scoped to frontend/**

The React plugin adds rules like "hooks must not be called conditionally" — rules that only make sense inside React components. If ESLint tried to apply those rules to your backend index.ts, it would throw false errors on perfectly valid Hono code.

The files key scopes a layer to matching paths only:

{
  files: ["frontend/**/*.{ts,tsx}"],
  plugins: { react, "react-hooks" },
  rules: { ... }  // only applied inside frontend/
}

Backend files are never touched by these rules. This is the monorepo payoff: one config file, different rules per package, no duplication.

Why the plugins live at the root

ESLint runs from wherever you invoke it — in this project, always from the repo root (npm run lint). When ESLint needs to load eslint-plugin-react, it looks for it relative to the process working directory, which is the root.

So all ESLint/Prettier packages belong in root devDependencies — even the React-specific ones — because the root is where ESLint runs. React runtime (react, react-dom) still lives in frontend/package.json because that's where it's bundled.

Check your understanding

1. Prettier reformats your code so all strings use double quotes. ESLint has a quotes: ["error", "single"] rule. What happens when you run both?

2. You move prettierConfig from last to first in your eslint.config.mjs array. What breaks?

3. You want ESLint to also check .js files in the backend. The current config only has files: ["**/*.{ts,tsx}"] for TypeScript rules. What do you add?

4. A teammate adds @typescript-eslint/no-explicit-any: "warn" after tseslint.configs.recommended in the config. Your existing "error" rule is in the layer directly below it. What does ESLint enforce?
