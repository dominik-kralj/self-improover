Hono + Neon + Drizzle

Your server, your database, your types — all three talking to each other.

The request-response cycle

Your frontend is code that runs in the browser. It can draw things on screen, but it has no way to store data that persists after the tab closes, or that other users can see. That's the job of a backend — a program that runs on a server (not in a browser), accepts requests from clients, and returns responses.

Every conversation between a frontend and a backend follows the same pattern:

Browser Server (Hono) ────── ───────────── 1. User submits a check-in form 2. Frontend sends an HTTP request ──► POST /checkins handler runs: validate, save to DB 3. Frontend receives response ◄── { id: 42, score: 3 } 4. UI updates

The request has two key parts: an HTTP method (GET, POST, PUT, DELETE) and a path (/checkins). Together they identify what the client wants to do. The server registers a route for each method + path combination, and runs the matching handler function when a request comes in.

Hono: an HTTP framework for TypeScript

You could write a raw Node.js server, but you'd spend most of your time parsing URLs, reading request bodies, and setting response headers by hand. A framework handles that plumbing so you only write the business logic.

We use Hono. It's tiny, runs on Node.js (and several other runtimes), and was designed for TypeScript from day one — every request and response is fully typed.

A minimal Hono app looks like this:

import { Hono } from "hono";
import { serve } from "@hono/node-server";

const app = new Hono();

app.get("/health", (c) => {
  return c.json({ status: "ok" });
});

serve({ fetch: app.fetch, port: 3000 });

app.get(path, handler) registers a route. The handler receives a context object (c) that wraps the request and lets you build a response. c.json() sends a JSON body with a 200 status.

Why Hono and not Express? Express is the classic Node.js framework, but it was written before TypeScript existed. Its types are bolted on as an afterthought. Hono was built for TypeScript: the context, request, and response are all precisely typed, so your editor catches mistakes before you run the code.

Postgres and Neon: where data lives

Your app's data — users, areas, commitments, check-ins — has to outlive any single request and be readable by future requests. It lives in a database.

We use PostgreSQL (Postgres), a relational database. Data is stored in tables with rows and columns, and tables can relate to each other. For example, a commitments row belongs to an areas row via a foreign key.

Running Postgres yourself means installing it, configuring it, keeping it updated, and backing it up. Neon takes care of all that: it's Postgres hosted in the cloud. You get a connection string — a URL that encodes everything needed to connect — and use it from your code.

postgresql://user:password@ep-something.us-east-2.aws.neon.tech/self_improover ──────────────────────────────────────────────────────────────────────────── │ │ │ │ │ protocol user password host database name

That string contains your database password. It must never appear in source code or get committed to git. More on that in the next section.

Drizzle ORM: your schema in TypeScript

You could query Postgres by writing raw SQL strings — but raw strings have no type checking. A typo in a column name only shows up at runtime, not at compile time.

An ORM (Object-Relational Mapper) lets you write queries in TypeScript instead of SQL. We use Drizzle, which takes a different approach from most ORMs: you define your schema as TypeScript objects, and Drizzle uses that to type every query automatically. 

import { pgTable, serial, text, integer } from "drizzle-orm/pg-core";

export const areas = pgTable("areas", {
  id:     serial("id").primaryKey(),
  name:   text("name").notNull(),
  weight: integer("weight").notNull().default(1),
});

This is the schema file. It's the single source of truth for what the areas table looks like. Drizzle derives TypeScript types from it automatically — no duplicate type definitions needed.

When you query this table, the result is fully typed:

const result = await db.select().from(areas);
// result is Area[] — TypeScript knows every column and its type
Schema first, types second. This matches the project's philosophy: Zod schemas are the source of truth for HTTP request/response shapes; Drizzle schemas are the source of truth for database shapes. TypeScript types are derived — never written by hand.

Environment variables: secrets don't belong in code

Your Neon connection string contains a password. If it appears in a file you commit, anyone with access to the repo — now or in the future — can connect to your database and read or delete all your data.

The solution is environment variables: values that exist outside your code, set in the environment where the process runs. In Node.js, you read them with process.env.VARIABLE_NAME: 

const connectionString = process.env.DATABASE_URL;
//    connectionString is string | undefined

During development, you store them in a .env file at the root of the package. A library called dotenv (or the newer built-in --env-file flag in Node 20+) reads that file and populates process.env before your code runs.

.env file (NOT committed) process.env (runtime, in memory) ───────────────────────── ──────────────────────────────── DATABASE_URL=postgresql://... ──► process.env.DATABASE_URL = "postgresql://..." PORT=3000 ──► process.env.PORT = "3000"

The .env file must be listed in .gitignore. You typically also commit a .env.example file with the same keys but empty values, so teammates know what to fill in.

The health check: one endpoint that proves everything works

When you start the server, two things need to be working: the server itself, and the database connection. A health check is a single endpoint — GET /health — that tests both and reports the result. 

app.get("/health", async (c) => {
  try {
    await db.execute(sql`SELECT 1`);   // cheapest possible DB query
    return c.json({ status: "ok" });
  } catch {
    return c.json({ status: "db_error" }, 500);
  }
});

If the server responds 200, you know: the process is running, process.env was loaded, Drizzle connected successfully, and at least one query completed. If it returns 500, you know the DB is unreachable — maybe the connection string is wrong, or Neon is down.

Health checks are used everywhere in production: load balancers hit them before routing traffic, monitoring services alert on non-200s, and deploy systems wait for a passing health check before marking a deploy successful.
